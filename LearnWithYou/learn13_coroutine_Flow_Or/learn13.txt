suspend

launch vs async

Trong bài này, bạn chỉ cần nắm chắc “đường ống” sau:

UI bấm nút → ViewModel launch → Repository gọi suspend api → Repository trả ApiResult → ViewModel update UiState → UI collect state và tự vẽ lại.

PHẦN A — suspend function: nó là “hàm có thể tạm dừng”

Hãy tưởng tượng bạn có dòng:

val dto = api.getUser(id)


Nếu getUser là hàm thường, nó phải trả dto ngay lập tức.
Nhưng gọi mạng thì không thể ngay lập tức.

Vì vậy Retrofit khai báo:

suspend fun getUser(id: Int): UserDto


Ý nghĩa cực chuẩn (không mơ hồ):

suspend = “hàm này có thể treo lại ở giữa chừng để chờ việc lâu (mạng/IO)”

nhưng khi “treo”, nó không block thread (không làm đứng UI)

và khi có kết quả, nó tiếp tục chạy từ đúng chỗ đó

Quy tắc vàng #1

suspend fun chỉ được gọi từ:

một suspend fun khác, hoặc

bên trong một coroutine (launch/async)

Nên nếu bạn viết trong ViewModel mà không launch, IDE sẽ chửi ngay.

PHẦN B — viewModelScope.launch: “khởi động một coroutine gắn với vòng đời ViewModel”

Bạn sẽ có kiểu:

fun loadUser(id: Int) {
    viewModelScope.launch {
        val result = repo.getUser(id) // repo.getUser là suspend
        // update state
    }
}

Vì sao phải launch?

Vì loadUser() là hàm thường (UI gọi vào), nó không phải suspend.
Nhưng nó cần gọi repo.getUser() (suspend).
→ bắt buộc phải “bọc” bằng coroutine.

viewModelScope là gì?

là scope do Android cung cấp cho ViewModel

khi ViewModel bị clear (màn hình đóng), nó tự cancel coroutine
→ tránh leak, tránh call xong UI đã chết.

Quy tắc vàng #2

launch dùng khi bạn muốn “bắn một công việc chạy nền” và không cần trả về giá trị ngay cho chỗ gọi.

Bạn gọi loadUser() từ UI, UI không cần nhận return trực tiếp; UI chỉ cần state được update.

PHẦN C — launch vs async: khác nhau đúng 1 thứ

Cùng là tạo coroutine, khác ở “có cần kết quả return không”.

launch

trả về Job

không có giá trị trả về

phù hợp: update UI state, call API xong set state

viewModelScope.launch {
    repo.getUser(id)
    // update state
}

async

trả về Deferred<T>

dùng khi bạn muốn lấy kết quả bằng await()

viewModelScope.launch {
    val deferred = async { repo.getUser(id) }
    val result = deferred.await()
}

Quy tắc vàng #3 (cho nhịp của bạn)

90% trường hợp ViewModel gọi Repository: dùng launch là đúng nhịp.
async chỉ dùng khi bạn cần chạy nhiều việc song song rồi await để gom kết quả.

PHẦN D — Flow/StateFlow: “dòng dữ liệu chảy”, UI chỉ việc collect

Ở bài 12–13, thứ bạn đang gặp thường là StateFlow (cho UI state).

Ví dụ ViewModel:

private val _uiState = MutableStateFlow(UiState())
val uiState = _uiState.asStateFlow()


Rồi khi có result:

_uiState.update { it.copy(isLoading = true) }
val result = repo.getUser(id)
_uiState.update { it.copy(isLoading = false, user = ...) }

Tại sao dùng Flow/StateFlow?

Vì UI cần “ngồi nghe” state thay đổi.

ViewModel phát state (emit/update)

UI collect state để render

Không còn kiểu UI đi hỏi repo liên tục.

PHẦN E — “collect Flow trong ViewModel” là khi nào?

Câu trong outline “collect Flow trong ViewModel” thường đúng ở 2 trường hợp:

Repo trả Flow (thường ở Room):

fun observeUsers(): Flow<List<User>>


→ ViewModel phải collect để chuyển thành UiState.

Bạn combine nhiều nguồn:

state UI

dữ liệu DB

network status

Nhưng ở nhịp hiện tại (chưa Room), bạn chỉ cần hiểu:

ViewModel update StateFlow

UI collect StateFlow

Chưa cần “collect trong ViewModel” nhiều.

3️⃣ Exception trong Coroutine – vì sao try/catch đặt ở Repository?

try {

    return ApiResult.Success(dto.toDomain())
} catch (e: IOException) {
    return ApiResult.Error("No internet")
}


Nhưng vì sao không catch ở ViewModel?

Lý do kiến trúc:

Repository = nơi giao tiếp dữ liệu thật

ViewModel = nơi quyết định state cho UI

👉 Nếu ViewModel tự catch:

nó sẽ phải biết lỗi mạng, lỗi server → bẩn kiến trúc

5️⃣ Structured Concurrency – lý do coroutine “không loạn”

1️⃣ Coroutine thật ra là gì? (nói thẳng)

Coroutine = một đơn vị công việc có thể tạm dừng và tiếp tục
chạy không block thread, và được quản lý theo cấu trúc cha–con.

👉 Nó KHÔNG PHẢI là:

thread

callback

magic async gì đó

👉 Nó là:

một “luồng logic” (logical flow)

được runtime chuyển qua lại giữa các thread khi cần

📌 Nói kiểu rất đời:

Coroutine giống như “việc đang làm dở, có thể bỏ đó đi chờ, rồi quay lại đúng chỗ”.

2️⃣ Coroutine có phải “phải có scope.launch { … }” không?

👉 Không hoàn toàn.
Nhưng trong Android thì 99% bạn sẽ thấy nó bắt đầu từ scope.launch.

Vì sao?

Vì coroutine luôn phải có nơi sinh ra (scope).

3️⃣ Định nghĩa CHÍNH XÁC một coroutine “đầy đủ”

Một coroutine đúng nghĩa luôn có 4 thành phần, dù bạn có thấy hay không:

Coroutine = Scope + Context + Job + Code


Giải thích từng cái (rất ngắn):

1️⃣ Scope – “ai quản lý nó?”

Ví dụ:

viewModelScope

lifecycleScope

Scope chết → coroutine chết

👉 Không có scope → coroutine không hợp lệ

2️⃣ Job – “nhiệm vụ này là gì?”

launch tạo ra 1 Job

Job có trạng thái:

active

cancelled

completed

👉 Khi bạn viết:

viewModelScope.launch { ... }


Thì:

bạn đã tạo 1 coroutine

bên trong có 1 Job

3️⃣ Context – “chạy ở đâu & theo luật gì?”

Context gồm:

Dispatcher (Main / IO / Default)

Job

Name (tuỳ chọn)

Exception handler

👉 Bạn thường không nhìn thấy, vì framework set giúp.

4️⃣ Code – “việc cần làm”

Là khối:

{
    val user = repo.getUser(id)
    _uiState.update { ... }
}

4️⃣ Vậy launch {} bên trong launch {} là gì?

Ví dụ:

viewModelScope.launch {
    launch {
        // việc A
    }
    launch {
        // việc B
    }
}


👉 Đây không phải là 1 coroutine duy nhất, mà là:

1 coroutine cha (outer launch)

2 coroutine con (inner launch)

Chúng tạo thành 1 cây coroutine:

viewModelScope
  └─ launch (cha)
       ├─ launch (con A)
       └─ launch (con B)


📌 Đây gọi là Structured Concurrency.

5️⃣ Một coroutine có thể tồn tại mà không có launch không?

👉 Có, nhưng bạn chưa cần dùng bây giờ.

Ví dụ:

suspend fun loadUser() {
    val user = repo.getUser(id)
}


👉 loadUser() KHÔNG phải là coroutine đang chạy
Nó chỉ là hàm “có thể được chạy trong coroutine”.

📌 Coroutine chỉ thực sự tồn tại khi:

có launch

hoặc async

hoặc withContext

6️⃣ Sơ đồ luồng chuẩn bạn đang dùng (Android thực tế)
UI click

👉 Coroutine chỉ được tạo ở ViewModel (hoặc UI)
👉 Repository không tạo coroutine, chỉ khai báo suspend.

7️⃣ Hiểu sai phổ biến (rất nhiều người mắc)
❌ “Coroutine = launch {}”

Sai.

❌ “Coroutine = async / await”

Sai.

❌ “Phải có nhiều launch lồng nhau mới gọi là coroutine”

Sai.

✅ ĐÚNG LÀ:

Coroutine = công việc async được quản lý bởi scope
launch chỉ là cách phổ biến nhất để tạo coroutine.

8️⃣ Câu chốt để bạn ghi nhớ (rất quan trọng)

Coroutine không phải là cú pháp.
Nó là một MÔ HÌNH THỰC THI.

Cú pháp (launch, suspend, async) chỉ là cách biểu diễn.
